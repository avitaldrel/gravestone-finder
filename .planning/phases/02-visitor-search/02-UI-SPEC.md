## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing, gap between result card icon and text |
| md | 16px | Default element spacing, card internal padding (px-4), result list gap |
| lg | 24px | Section padding, gap between search bar and results area |
| xl | 32px | Hero section vertical padding on mobile |
| 2xl | 48px | Hero section vertical padding on desktop |
| 3xl | 64px | Top-of-page spacing on desktop (search bar vertical centering offset) |

Exceptions: Touch target minimum 44px height for the search input on mobile (accessibility). The shadcn Input component default height of `h-10` (40px) must be overridden to `h-12` (48px) for the hero search bar.

**Source:** Tailwind default spacing scale (multiples of 4). Existing admin page uses `px-4 py-8`, `gap-6` (24px) -- this phase follows the same baseline.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` / oklch(1 0 0) | Page background, search section background |
| Secondary (30%) | `var(--card)` / oklch(1 0 0) with `ring-1 ring-foreground/10` | Result cards, not-found container, no-data container |
| Accent (10%) | `var(--primary)` / oklch(0.205 0 0) | Search input focus ring only |
| Destructive | `var(--destructive)` / oklch(0.577 0.245 27.325) | Not used in this phase (no destructive actions) |
| Muted foreground | `var(--muted-foreground)` / oklch(0.556 0 0) | Location text, placeholder text, guidance text, MapPin icon |

Accent reserved for:
- Search input focus ring (`focus-visible:border-ring focus-visible:ring-ring/50`)
- "View full directory" link text (`text-primary`)

**What accent is NOT used for in this phase:** No primary-colored buttons, no accent backgrounds, no colored badges. The visitor search page is intentionally quiet and neutral -- the content (veteran names and locations) is the focus, not the UI chrome.

**Source:** `globals.css` `:root` color tokens. The neutral base color (all oklch values with 0 chroma) means the design is monochromatic by default. This is appropriate for a memorial/respectful context.

## Layout Contract

### Page Structure (D-01, D-03, D-15)

The root page (`/`) is a single-column, vertically centered hero layout.

```
+------------------------------------------+
|                                          |
|          [welcome heading]               |
|          [welcome subtitle]              |
|                                          |
|     [==== search input ====]             |
|                                          |
|     [result card 1]                      |
|     [result card 2]                      |
|     [result card 3]                      |
|          ...                             |
|                                          |
+------------------------------------------+
```

**Container:** `max-w-2xl mx-auto px-4 sm:px-6` (narrower than admin's `max-w-3xl` -- search results are single-column cards that read better at narrower width).

**Vertical centering:** Before the user types, the welcome + search bar should sit in the upper-middle of the viewport. Use `pt-16 sm:pt-24 md:pt-32` to push content down without true vertical centering (which would cause layout shift when results appear below).

**Results area:** Directly below the search bar with `mt-6` (24px) gap. Results stack vertically with `gap-3` (12px) between cards.

### Result Card Layout (D-05, D-06)

Each result card uses the existing `Card` + `CardContent` components.

```
+------------------------------------------+
| [Veteran Name]                 [MapPin]  |
| Row B, Position 12                       |
+------------------------------------------+
```

- Card uses `size="sm"` variant for compact display
- `flex items-start justify-between` for name + icon row
- Veteran name: `font-medium` (Label role)
- Location: `text-sm text-muted-foreground` (Body role)
- MapPin icon: `h-5 w-5 text-muted-foreground`, top-right corner
- No card border highlight or hover effect -- these are read-only display cards

### Search Input (D-02, D-03)

- Use shadcn/ui `Input` component (add via `npx shadcn@latest add input`)
- Override height to `h-12` (48px) for hero prominence and mobile touch target
- Override text size to `text-base` (16px) to prevent iOS zoom on focus
- Full width within the container: `w-full`
- Search icon (lucide `Search`) inside the input as a left adornment: `pl-10` padding + absolutely positioned icon
- Placeholder text: "Search by veteran name..."
- No submit button (D-02: instant results as user types)
- `type="search"` for native clear button on mobile browsers
- `autoComplete="off"` to prevent browser autocomplete interference
- `autoFocus` on desktop; no autofocus on mobile (keyboard covers content)

## State Machine

Four mutually exclusive visual states (from RESEARCH.md):

| State | Condition | What Renders |
|-------|-----------|-------------|
| **no-data** | `flags.length === 0` | No-data message (search bar hidden per D-14) |
| **idle** | `flags.length > 0 && query === ""` | Welcome message + search bar, nothing below (D-15) |
| **results** | `flags.length > 0 && query !== "" && results.length > 0` | Welcome message + search bar + result cards |
| **not-found** | `flags.length > 0 && query !== "" && results.length === 0` | Welcome message + search bar + not-found guidance |

No loading state needed -- Fuse.js search is synchronous and instant for ~100 records. Flag data is fetched server-side (no client fetch waterfall).

## Interaction Contract

### Search Input Behavior (D-02, D-08, D-09, D-10)

| Interaction | Behavior |
|-------------|----------|
| User types | Debounce 200ms, then run Fuse.js search |
| Minimum query length | 2 characters (`minMatchCharLength: 2` in Fuse config) |
| 1 character typed | No results shown (treated as idle) |
| Results order | Best match quality first (Fuse.js `shouldSort: true`) |
| Clear input | Return to idle state (just search bar, no results) |
| Browser back/forward | No URL state for search query in v1 (revisit in Phase 3 if needed) |

### Accessibility

| Requirement | Implementation |
|-------------|---------------|
| Search input label | `<label>` element with `sr-only` class, text: "Search for a veteran's flag" |
| Results announcement | `aria-live="polite"` region wrapping the results area. Screen reader hears "{N} results found" or "No flags found" |
| Result cards | Semantic `<article>` or Card `<div>` with clear heading hierarchy |
| Keyboard navigation | Tab moves through interactive elements. No custom keyboard shortcuts needed. |
| Color contrast | All text meets WCAG 2.1 AA. `muted-foreground` oklch(0.556 0 0) on white background = ~4.7:1 ratio (passes AA for normal text) |

### Responsive Behavior

| Breakpoint | Layout Change |
|------------|---------------|
| < 640px (mobile) | `px-4` container padding, `pt-16` top spacing, search input `text-base` (prevents iOS zoom) |
| >= 640px (sm) | `px-6` container padding, `pt-24` top spacing |
| >= 768px (md) | `pt-32` top spacing (pushes hero further down for more dramatic centering) |

No horizontal layout changes. The single-column design works identically across breakpoints -- only vertical spacing adjusts.
