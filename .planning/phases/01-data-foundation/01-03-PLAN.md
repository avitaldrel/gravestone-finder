    Create `src/app/api/flags/import/route.ts` as a Next.js App Router API route handler.

    **POST /api/flags/import**

    Request body shape:
    ```typescript
    {
      flags: FlagInsert[];     // Array of validated flag data
      mode: "replace" | "merge"; // Import strategy chosen by user
    }
    ```

    Implementation:

    1. Import `createClient` from `@/lib/supabase/server` and `FlagInsert` from `@/lib/types/flag`.

    2. Export an async `POST` function that:
       a. Parses the request body as JSON
       b. Validates that `flags` is an array and `mode` is "replace" or "merge"
       c. If `mode === "replace"`:
          - Delete all existing flags: `supabase.from("flags").delete().gte("id", 0)`
          - CRITICAL: Supabase requires a filter on delete. Use `.gte("id", 0)` to match all rows.
          - If delete errors, return `NextResponse.json({ error: "Failed to clear existing data" }, { status: 500 })`
          - Insert new flags: `supabase.from("flags").insert(flags).select()`
       d. If `mode === "merge"`:
          - Upsert flags: `supabase.from("flags").upsert(flags, { onConflict: "row_label,position" }).select()`
       e. Return `NextResponse.json({ count: data.length })` on success
       f. Return `NextResponse.json({ error: error.message }, { status: 500 })` on failure

    3. Also export an async `GET` function to fetch all flags:
       - `supabase.from("flags").select("*").order("name")`
       - Return `NextResponse.json(data)`
       - This is used by the admin page to check if data exists (for D-13 dialog trigger) and to display the imported data table.

    4. Also export an async `DELETE` function to support data clearing:
       - `supabase.from("flags").delete().gte("id", 0)`
       - Return `NextResponse.json({ success: true })`

    5. Import `NextResponse` from `next/server`.

    Use the Supabase server client (not browser client) since this runs server-side in a route handler.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> API route | Client sends parsed flag data as JSON to /api/flags/import |
| API route -> Supabase | Server-side route handler writes to database |
| User file -> Browser | Untrusted file processed client-side by SheetJS + Zod (handled in Plan 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Tampering | /api/flags/import POST | mitigate | Validate request body shape: check `flags` is array, `mode` is "replace" or "merge". Supabase JS client uses parameterized queries -- no SQL injection risk. |
| T-01-09 | Denial of Service | drop-zone.tsx | mitigate | File size limit at 5MB via react-dropzone `maxSize`. File type restricted to spreadsheet formats via `accept` filter. |
| T-01-10 | Spoofing | /api/flags/import | accept | No auth (D-04). Anyone can POST to the import endpoint. This is by design for this event app. If abuse occurs, add auth in a future phase. |
| T-01-11 | Tampering | XSS via flag name data | mitigate | React auto-escapes all JSX output. Flag names displayed via React components, never via `dangerouslySetInnerHTML`. Zod trims name strings. |
</threat_model>
